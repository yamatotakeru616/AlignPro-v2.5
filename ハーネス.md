# ハーネス - 3D CIM道路アライメント動作検証仕様書

本仕様書は、3D CIM道路アライメント設計ツールの各機能が正常かつ正確に動作しているか、および例外操作時の頑健性（ロバストネス）を検証するためのテスト手順、境界条件、および判定基準を定めたものです。

---

## 1. 自動/手動検証テストケース（正常系）

### 1.1 テストケース 01: MAP (平面線形) ドラッグ＆ドロップ検証
- **操作手順**:
  1. 「MAP」タブを選択し、初期状態の3点（BP, IP, EP）を確認する。
  2. 交点（IP）のドラッグハンドルを指、またはマウスで長押しし、任意の方向へ30px以上移動させる。
- **期待される挙動**:
  - コントロールポイントの座標（X, Y, Z）が即座に追従して更新されること。
  - BP → IP → EP を結ぶ破線と、補間された滑らかな道路中心線（実線）が、遅延なくリアルタイムに描画更新されること。
  - 更新後の座標変化が、他のすべてのタブ（「Drawings」、「3D Preview」）へ即座に反映されていること。

### 1.2 テストケース 02: DRAWINGS (図面) 縦横断面とスライダー検証
- **操作手順**:
  1. 「DRAWINGS」タブを選択。
  2. スライダーを動かして「左車線幅 (Left Lane Width)」をデフォルトの `3.25m` から最大値の `5.0m` へ変更する。
  3. 「法面勾配 (Slope Gradient 1:S)」を `1.5` から `2.5` へ変更する。
- **期待される挙動**:
  - 横断図のSVG描画が即座に横方向に伸長し、法面の傾斜が緩やか（1:2.5のなだらかな傾斜）に変化すること。
  - 計画高が地盤高より高い断面では、赤い斜線/塗りの「盛土 (Fill)」ハッチングが正確に描画されること。
  - 計画高が地盤高より低い断面では、青い斜線/塗りの「切土 (Cut)」ハッチングが正確に描画されること。

### 1.3 テストケース 03: 3D PREVIEW (立体可視化) 連動検証
- **操作手順**:
  1. 「3D PREVIEW」タブを選択。
  2. 画面を1本指でスワイプ（またはマウス左ドラッグ）してカメラを回転させ、2本指ピンチ（またはマウスホイール）でズームする。
  3. コントロールパネルの「右車線幅」スライダーを最小の `2.0m` に動かす。
- **期待される挙動**:
  - 道路の3Dモデル（アスファルト部分）が即座にスリムに変形すること。
  - カメラの回転やズーム操作が引っかかりなく滑らか（秒間60フレーム目安）に行えること。
  - タブを切り替えて「MAP」などに戻ったとき、3Dのレンダリングループ（`requestAnimationFrame`）が自動的に一時停止し、CPU/GPUに不必要な負荷をかけないこと。

### 1.4 テストケース 04: EXPORT (成果物出力) 検証
- **操作手順**:
  1. 「EXPORT」タブを選択。
  2. 現在設計された線形に適合した「LandXML」および「JSON」コードが表示されていることを確認する。
  3. 「LandXMLをコピー」ボタンをクリックする。
- **期待される挙動**:
  - クリップボードにXMLテキストが保存され、メモ帳などにペーストできること。
  - 「XMLをダウンロード」をクリックした際、`cim_road_alignment.xml` というファイル名でダウンロードダイアログがトリガーされること。

## 2. 異常系・境界条件テストケース（耐久性検証）

### 2.1 制御点の極端な接近・交差
- **条件**: 交点（IP）を、始点（BP）または終点（EP）の真上（距離5m未満）にドラッグする。
- **期待される挙動**:
  - 幾何学的エラーや無限ループによるブラウザフリーズが発生しないこと。
  - 最小近接境界（例えば20m）に自動的に吸着（クリップ）され、道路の曲率が破綻しないガードが機能すること。

### 2.2 スライダー値の最小・最大限界
- **条件**: スライダーパラメータの範囲外への不正なキー入力、または操作時のガード。
- **期待される挙動**:
  - 車線幅が0やマイナス、法面勾配が0やマイナスなどの無効値に設定されないよう、入力値が `Math.max(min, Math.min(max, value))` によって完全に安全域にロックされていること。

### 2.3 WebGL 複数コンテキスト型競合の防止（新規追加）
- **条件**: 3Dビューポートを表示する際、ブラウザの特定プラグインや事前診断スクリプトが同一 Canvas 上で `getContext('webgl')` (WebGL1) を取得し、その後に Three.js が `WebGL2` を要求する環境。
- **期待される挙動**:
  - Canvas 上で直接診断を走らせる代わりに隔離された `dummyCanvas` 上で事前判定を行うため、本番 Canvas に「Canvas has an existing context of a different type」エラーが発生せず、クリーンに WebGL2 コンテキストが生成されること。
  - もしデバイス全体の制限により WebGL 自体が初期化不可能な場合、白濁したブラウザクラッシュ（Uncaught Exception）を起こさずに、美しく安全な「WebGL が無効か非対応です」警告画面にフォールバックすること。
