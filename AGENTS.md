# プロジェクト開発ルール

## ドキュメントのリアルタイム自動更新
- 本プロジェクトで新機能の追加、仕様変更、技術的なバグ修正、または機能の改善を行った場合は、**必ずその都度 `/機能一覧書.md` を最新の状態に更新・追記すること**。
- `/機能一覧書.md` には「実装済み機能」「実施した技術的改善実績」「今後の追加案」が体系的に整理されているため、この構成を崩さず、常に整合性が保たれた状態を維持すること。

## モバイル・低スペック端末タッチ操作の自動回帰チェックリスト
平面マップ（MapTab）、図面（DrawingsTab）、3Dビュー（ThreeCanvas）における操作機能の追加やリファクタリングを行う際は、モバイル・低スペック端末での操作性および性能確保のため、以下のチェック項目に反していないかを必ず検証すること。

### 1. Pointer Event と Pointer Capture の整合性
- **チェック項目**: 平面アライメント（IP点）や縦断面計画高などのドラッグ操作時、対象のコントロールポイントがポインターキャプチャ（`setPointerCapture` および `releasePointerCapture`）を正しくハンドリングしているか。
- **検証項目**: タッチパネルでのスクロール等と競合しないよう、操作対象のキャンバスおよびドラッグ要素に CSS で `touch-action: none`（または部分的な制限）が正しく指定されていること。

### 2. タッチターゲットサイズ（最小 44px）の確保
- **チェック項目**: すべてのタップ可能なUIコンポーネント（フローティング操作盤、設定変更ボタン、タブ切り替えなど）は、タッチ可能領域が 44px × 44px 以上確保されているか。
- **検証項目**: ビジュアルサイズが小さいドラッグ用のハンドル円（IP点など）であっても、見えない透明なタッチ用バウンダリ（SVGの透明なオーバーレイサークルや、CSSの拡張パディング）を広げることで、タッチがすり抜けないよう工夫されていること。

### 3. マルチタッチ（ピンチズーム・パン）の完全分離
- **チェック項目**: 2本指でのピンチズーム操作時に、1本指でのアライメント引きずり（ドラッグ）や単発パンと干渉していないか。
- **検証項目**: ポインター数（ポインターイベントの `pointerId` 追跡）を管理し、マルチタッチ時は線形変形ロジックをバイパスして誤作動を防ぐ防衛ガードが入っていること。

### 4. フローティングUIの狭小画面ガードとドラッグ負荷削減
- **チェック項目**: 画面外へのクランプ処理が正しく働き、モバイルなどの低解像度画面でフローティングUIが画面外にクリップされて操作不能・紛失状態にならないか。
- **検証項目**: ドラッグ移動中の位置更新（State更新）による再描画負荷を最小限に抑え、低スペック端末のVRAM4GB制限下でのブラウザのフレームレート低下を防止していること。

### 5. ポインターキャンセルのリセット保証
- **チェック項目**: `pointerup` だけでなく `pointercancel` イベント（画面外へのドラッグ、OSのシステムポップアップや通知の割り込み、ジェスチャーの開始）に対しても、確実にドラッグ状態・パン状態が初期化（リセット）される設計になっているか。
