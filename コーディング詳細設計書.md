# 3D CIM道路線形プロトタイプ コーディング詳細設計書

本設計書は、GitNexusコーディング開発憲法に基づき、3D CIM道路線形（アライメント）プロトタイプの画面仕様、状態管理、Androidでの操作性、および例外時の挙動を定義するものです。

---

## 1. 画面仕様（UI/UX設計）

本アプリは、モバイル（特にAndroid端末）およびPC（3050Ti環境）で高速に動作する、ダークモードを基調としたグラスモーフィズムデザインのシングルビュー構造を採用します。画面上部には概要、中央には各機能（4つのタブ）、下部にはコントロールパネルを配置します。

### 1.1 4つのタブビュー構成

1. **「MAP (地図)」タブ**
   - **機能**: MapLibre GLを利用して、簡易的な地表等高線（等高線風のグラデーションまたは標高シミュレーション）の上に、道路の始点（BP: Beginning Point）、中間交差点（IP: Intersection Point）、終点（EP: Ending Point）を配置し、ラインで結びます。
   - **インタラクション**: 各コントロールポイント（BP, IP, EP）はマップ上でドラッグ＆ドロップにより移動可能です。移動に追従して、道路の平面線形（直線、IP部曲線のクロソイド風ベジエ補間）がリアルタイムに再計算され描画されます。

2. **「DRAWINGS (図面)」タブ**
   - **縦断図 (Profile View)**: 道路の横軸（追加距離/Station）に対する縦軸（標高/Elevation）を表示するSVG描画。計画高（Red line）と地盤高（Gray dashed line）を対比し、計画高が地盤高より高ければ赤色の「盛土 (Fill)」、低ければ青色の「切土 (Cut)」のハッチングを重ねて描画します。
   - **横断図 (Cross-Section View)**: 選択した代表地点での道路の横断形状を示すSVG描画。
     - スライダーで調整可能なパラメータ:
       - 左車線幅 (Left Lane Width): 2.0m 〜 5.0m (デフォルト: 3.25m)
       - 右車線幅 (Right Lane Width): 2.0m 〜 5.0m (デフォルト: 3.25m)
       - 路肩幅 (Shoulder Width): 0.5m 〜 2.5m (デフォルト: 1.0m)
       - 法面勾配 (Slope Gradient 1:S): 1:0.5 〜 1:3.0 (デフォルト: 1:1.5)

3. **「3D PREVIEW (3Dプレビュー)」タブ**
   - **機能**: Three.jsのCanvasを利用して、平面線形（Map）および縦断線形、横断面パラメータ（Drawings）から動的にスイープ（掃引）生成された3D道路メッシュをリアルタイムに3D描画します。
   - **可視化要素**: アスファルト路面（ダークグレー）、路肩（グレー）、法面（切土時は青、盛土時は赤のサーフェス）、地盤（グリーン平面またはワイヤーフレーム地形）。
   - **操作**: タッチ操作による回転（1本指ドラッグ）、ズーム（ピンチイン・アウト）、パン（2本指ドラッグ）に対応します。

4. **「EXPORT (書き出し)」タブ**
   - **機能**: 現在設計中の線形データおよび道路断面パラメータを、道路設計の標準フォーマットである「LandXML」のモック表現、およびシステム連携用の高精度JSON形式で書き出します。
   - **インタラクション**: ワンクリックでクリップボードへコピーするボタン、および `.xml` / `.json` ファイルとしてローカルにダウンロードするボタンを設置します。

### 1.2 グラスモーフィズムUIとコントロールパネル
- **デザイン**: 深いチャコールグレー（`bg-slate-900/80`）をベースに、ぼかし（`backdrop-blur-md`）、細い白ボーダー（`border-white/10`）を適用した近未来的なグラスモーフィズムデザイン。
- **パネル制御**: コントロールパネルは右側または下部に配置し、1タップで最小化（Collapse）/展開（Expand）が可能なトグルボタンを提供。
- **詳細設計書表示**: UI上に「詳細設計書」をフローティングダイアログまたは引き出し型パネルで常時閲覧できるボタンを配置。
- **テキストコピー**: 開発者向けに「ライブプレビューテキストをコピー」ボタンを設置し、現在のパラメータ一式を即座にコピー可能にします。

## 2. 状態管理（State Management）

複雑な3D連動をシームレスかつ軽量に行うため、Reactの最適化された `useState` / `useMemo` / `useRef` を組み合わせた単一データソース（Single Source of Truth）による状態管理を構築します。Android端末（VRAM 4GB、3050Ti等）の制約を考慮し、不必要な再レンダリングを抑止します。

```typescript
// 状態の構造定義
interface ControlPoint {
  id: 'BP' | 'IP' | 'EP';
  x: number; // 平面座標X (m)
  y: number; // 平面座標Y (m)
  z: number; // 標高Z (m)
}

interface CrossSectionParams {
  leftLaneWidth: number;   // 左車線幅 (m)
  rightLaneWidth: number;  // 右車線幅 (m)
  shoulderWidth: number;   // 路肩幅 (m)
  slopeGradient: number;   // 法面勾配 S (1:S)
}

interface AlignmentState {
  points: ControlPoint[];          // 制御点配列
  crossSection: CrossSectionParams; // 横断パラメータ
  selectedStation: number;         // 横断図を表示する現在地点 (0.0〜1.0)
  activeTab: 'map' | 'drawings' | '3d' | 'export';
  panelCollapsed: boolean;         // コントロールパネルの最小化状態
  showSpecs: boolean;              // 設計書ダイアログの表示状態
}
```

### 2.1 線形のリアルタイム補間ロジック
- **平面線形**: BP → IP → EP の折れ線をベースに、IP部で滑らかに曲がるように、2次ベジエ曲線（Quadratic Bezier）または円弧を用いた補間を適用し、等間隔（例：5mピッチ、計50点程度）の計画線座標を `useMemo` で軽量に事前計算。
- **縦断線形**: 各計画線の位置において、地盤高（数式で定義されるモック地形：例：`Z = sin(X/100)*cos(Y/100)*15 + 10`）と計画高（BPのZ、IPのZ、EPのZを線形または放物線補間した値）を動的に算出。

## 3. Androidおよびスマホでの操作性・最適化

Android等のタッチデバイス環境で快適に利用できるよう、以下のモバイルファースト設計を行います。

1. **タッチターゲットの確保**:
   - 地図上のBP, IP, EPのドラッグ操作用アンカー、および各種タブ切り替え、最小化ボタンは、指での操作を考慮し、最低 **44px × 44px** 以上の有効クリック・タッチ領域を確保します。
2. **タッチドラッグの干渉防止**:
   - 地図上のコントロールポイントのドラッグ操作時に、ブラウザ全体のスクロールやスワイプバックが誤爆しないよう、`touch-action: none` 属性をドラッグ可能要素に適用します。
3. **Three.jsのタッチ操作最適化**:
   - Three.jsでのカメラコントロール（OrbitControlsに類似した自作のシンプルなジェスチャー制御、または軽量OrbitControls）は、1本指でのスワイプ回転、2本指ピンチでのズームをサポートし、滑らかな慣性移動を最小限のコードで実現。
4. **低メモリ/低描画負荷（PC 3050Ti / Android 4GB VRAM）対策**:
   - Three.jsのレンダリングループ（`requestAnimationFrame`）は、**「3D Preview」タブがアクティブな時のみ実行**させ、他のタブ表示中は描画を停止し、バッテリー消費とCPU/GPU負荷を極限まで低減します。
   - 頂点数は必要最小限（道路断面50点 × 断面10点 ＝ 500ポリゴン程度）に抑え、リソースを無駄に消費しないようにします。

## 4. 例外時の挙動とエラーハンドリング

1. **制御点の逆転・極端な位置のガード**:
   - IPがBPよりも手前に引きずられたり、EPが極端にBPに近づきすぎた場合、道路の破綻（特異点）を防ぐため、ポイント間の最小距離（例: 20m）を設定し、それ未満にならないようドラッグ時の座標を補正します。
2. **WebGL非対応・初期化失敗時のフォールバックとコンテキスト重複防止**:
   - 実際の描画用 `canvas` に対して事前に `getContext` を呼び出してテストすると、ブラウザが WebGL1 にコンテキストを固定してしまい、Three.js の WebGL2 初期化時に「Canvas has an existing context of a different type」エラーでクラッシュする原因になります。
   - そのため、判定用の **別ダミーキャンバス (`document.createElement('canvas')`)** をメモリ上で作成し、そちらで WebGL/WebGL2 のコンテキスト確認を実施することで本番キャンバスをクリーンに保ちます。
   - さらに `new THREE.WebGLRenderer` 呼び出し部分を `try-catch` ブロックで安全に保護し、万が一の初期化例外時にも画面上に「WebGL が無効か非対応です」という日本語警告と再読み込み用の復旧ボタンを表示してフェイルセーフにします。
3. **MapLibre GLのフォールバック**:
   - オフライン環境や地図タイル取得失敗時でも、等高線グリッドとコントロールポイントのドラッグ機能自体はSVG/Canvasベースの「ベクトル代替マップ」として正常に動作するよう二重のガードを設けます。

## 5. ドキュメント自動同期とループ改善

- **ドキュメントセット運用**: 実装コードを変更する際は、必ず本 `コーディング詳細設計書.md` および `プロジェクト内容設計書.html` を同時に更新します。
- **エラー修正ループ**: コンパイルエラーや動作上の不具合を検知した場合、即座に修正を繰り返し、エラーログが完全にゼロ（DISMISS ALLエラー無し）の状態を維持します。
- **天才クロちゃんへのエスカレーション**: もしどうしても解決できない特殊な環境依存バグや技術的壁に直面した場合は、勝手にダミー実装でお茶を濁さず、即座に `/助けてクロちゃん.md` を作成してオーナーの指示を仰ぎます。
